# 自动化广撒网抓取方案

这套方案的目标不是一开始就判断哪个链接有用，而是先把候选页系统化抓下来，再做筛选和清洗。

## 两阶段流程

1. `discover_candidate_sources.py`
   - 输入 `configs/discovery_seeds.csv`
   - 按种子站点和允许域名做小范围广撒网
   - 把访问过的页面缓存到 `raw_data/discovery/`
   - 产出：
     - `reports/discovery_pages.csv`
     - `reports/discovery_candidates.csv`

2. 候选筛选
   - 按 `score`、`matched_keywords`、`target_suffix` 筛选高价值链接
   - 优先保留：
     - `招生计划`
     - `历年分数 / 往年分数 / 投档分`
     - `招生章程`
     - `专业目录 / 专业介绍`
     - `广西 / 物理 / 本科`

## 使用方式

先跑种子发现：

```bash
python3 scripts/discover_candidate_sources.py --insecure
```

如果只想先试一个站点：

```bash
python3 scripts/discover_candidate_sources.py \
  --seed-id bupt_official \
  --seed-id xidian_official \
  --insecure
```

## 推荐实践

- 本地网络不稳定时，优先在服务器跑，再把 `raw_data/discovery/` 和 `reports/discovery_*.csv` 同步回来。
- 先把候选池做大，再逐步收紧过滤规则，不要反过来。
- `CHSI` 这类会返回 `412` 的站点保留在种子里，后面可以接浏览器登录态或更完整请求头继续补抓。
- 不把非官方站点当主源；第三方镜像最多作为补链入口。

## 效率优先的执行规则

自动化与手工推进都遵循同一套执行规则：

1. 每轮必须至少落地 5 所学校的真实新增数据。
   - 优先统计已写入本地结构化表或报告的新增学校。
   - 仅有发现页、入口页、接口探测结果，不算一轮完成。

2. 候选来源按收益排序推进。
   - 第一优先：已缓存官方 HTML/PDF/JSON。
   - 第二优先：公开 API。
   - 第三优先：静态详情页。
   - 第四优先：403/协议阻塞/需要会话模拟的站点。

3. 对 403、网络不通、表单回放失败的学校，只做一次有效定位后转入阻塞待办。
   - 不在同一轮里反复尝试多个高阻塞站点。
   - 除非有新的低成本突破口，否则主线优先推进能直接产出结构化数据的学校。

4. 每轮结束后统一顺序刷新：
   - `seed -> school_summary -> structured_coverage -> target_coverage -> pipeline_status -> probe_backlog`
   - 不并行刷新这些汇总脚本。

5. 学校级摘要层允许使用“官方专业分明细派生摘要”，但必须保留标签说明。
   - 要明确写明“由官方专业分明细推导”，不能冒充原始学校级公告。

6. 防拖挂规则优先于“看起来更快”的并行技巧。
   - 本地检查、状态核对、文件抽样、覆盖率验证默认串行执行。
   - 不把多个短检查放进同一组并行批处理，避免单个子命令拖住整轮。
   - 先生成结构化结果，再单独跑最短的核对命令；核对失败不回滚已落地成果。

7. 所有非核心探测都必须可被快速截断。
   - 递归搜索、长脚本扫描、接口试探只允许在很小范围内执行。
   - 一旦发现结果回传异常慢、没有新数据、或只是为了“再确认一下”，就直接停止。
   - 自动化更看重稳定交付，不追求一次性把所有检查做完。
